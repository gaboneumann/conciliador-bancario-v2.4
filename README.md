# 🏦 Bank Reconciliation V2

> Bank reconciliation system built in Python, shaped by the real-world context of an SII audit accountant.
> The program compares the bank statement against the auxiliary ledger, classifying each transaction into 3 match levels and generating 3 reports.
> The Excel files come out audit-ready — including a critical-findings dashboard with validated mathematical balancing.
> ⚠️ All data is 100% synthetic, generated by an external script.

## The Problem

Bank reconciliation is a repetitive, manual process in the auditing world — highly prone to human error and costly in time for companies and SMBs. This system automates the entire process: from reading Excel files to generating audit-ready reports, with configurable tolerances, automatic diagnosis of discrepancies, and full mathematical balancing.

**Solution:** a configurable Python pipeline that reads the two Excel files, matches every transaction across three certainty levels, and emits three audit-ready reports with total mathematical balancing.

## Configuration

All tolerances are configurable in `config/config.py`:

```python
TOLERANCIA_MONTO_PCT     = 0.02    # ±2% amount difference
TOLERANCIA_MONTO_ABS_MAX = 5_000   # absolute cap in CLP
TOLERANCIA_DIAS          = 5       # ±5 days date difference
TOLERANCIA_REFERENCIA    = 6       # first 6 characters of the reference
FACTOR_IVA               = 1.19    # ratio for net vs. gross detection
```

## Core Mechanism

For each bank-statement transaction, the ledger is searched in priority order:

```
Hierarchy: RUT → Amount → Date → Reference

1. Exact     → RUT + amount ±2% + date ±5 days + reference (6+ chars)
2. Suggested → RUT + amount ±2% + date ±5 days (or IVA difference ×1.19)
3. Manual    → no match — automatic diagnosis of the reason
```

Each ledger transaction can be used only once, preventing duplicate matches. The index by RUT reduces complexity from O(n²) to O(n).

### Key Components

- **Index by RUT**: groups ledger entries by RUT so each statement transaction is matched in O(n) instead of O(n²).
- **Single-use ledger entries**: a ledger transaction is consumed once matched, avoiding duplicate matches.
- **Automatic diagnosis**: when no match is found, the reason is classified automatically.

**Automatic diagnosis of unreconciled items:**

| Reason | Meaning |
| :--- | :--- |
| Date matches but amount not found | A movement exists on that date but the amount is very different |
| Amount matches but date out of range | The transaction exists but the offset is > 5 days |
| Possible Net vs. Gross (×1.19) | Exact IVA difference — Suggested match |
| Transaction absent from the auxiliary ledger | No similar record exists |

## Generated Reports

### 1. `conciliacion_resultado.xlsx`

All transactions with their match. Columns: bank-statement data, matched ledger data, match type, certainty, differences, flags, and aging.

### 2. `partidas_sin_conciliar.xlsx`

The 123 transactions with no counterpart in the ledger, with the diagnosed reason and the closest amount found.

### 3. `hallazgos_criticos_auditoria.xlsx`

Ranking by RUT with total mathematical balancing. Includes three impact families:

| Family | MI | Traceability ID |
| :--- | :--- | :--- |
| Manual | `monto_cartola` | `nro_documento` |
| Suggested | `monto_cartola - monto_libro` | `nro_documento` |
| Ledger without counterpart | `-monto_libro` | `nro_comprobante` |

`sum(MI) == saldo_cartola - saldo_libro`

**Color logic:**

- Red background `#FFC7CE` — the RUT concentrates more than 20% of the total error
- Orange text `#9C5600` — aging greater than 90 days
- **Bold** — UNIDENTIFIED RUT (top priority)

## Design Principles

- **Separation of concerns**: each module has a single responsibility and can be modified without affecting the rest.
- **Configurable tolerances**: amount, date, reference, and IVA thresholds all live in `config/config.py`.
- **TDD first**: tests are written before production code (265 tests).
- **O(n) performance**: the index by RUT keeps matching linear instead of quadratic.
